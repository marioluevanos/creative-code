Resource Article Updates with ISR

Context

We are nearing the Resource article release

  1. To rebuild all 500+ articles each time a build is triggered
  2. Limit this to the latest 100 articles and have the rest built per request.
Describe details of Component Library
  • thin and light borders
  • open and closed states
  • input fields
  • day night play badges

Decision Drivers

A Next.js configuration that will allow the Resource articles to scale in volume and that this solution would not negatively affect the build and deployment times.

Considered Options

  • Server-side Rendering (SSR)
  • Static Site Generation (SSG)
  • Incremental Static Regeneration (ISR)

Server-side Rendering (SSR)

Positive

  • The data used to generate the page is fetched at the request time.
  • The HTML is generated on each request.
  • Access to URL parameter to handle dynamic pages

Negative

  • The first initial load can take a long time.
  • If APIs are slow, this can negatively impact the drawn-out rendering cycle.
  • The app deals with Hydration as overhead for the rendering cycle.
  • Need to set caching via request headers manually.
